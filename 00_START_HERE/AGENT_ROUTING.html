Course-specific answer rules (apply when writing homework/exam-style answers)

  • Dynamic Programming — structure every answer as: (1) subproblem in words, (2) table entry + index ranges, (3) base cases, (4) recurrence with each min/max option explained, (5) fill order, (6) where the final answer lives, (7) runtime = #entries × work-per-entry. Course DP answers are bottom-up, not recursion/memoization sketches. See DP guidance and DP recurrences.
  • Divide & Conquer — narrative algorithm / correctness / runtime. Use words, not pseudocode, when the prompt asks for an algorithm. See D&C guidance.
  • Graphs & Flows — solve by transforming input/output around an allowed black box (BFS, DFS, Dijkstra, SCC, MST, max-flow). Do not rewrite a black box's internals. Use only the algorithms in usable black boxes.
  • Runtimes — worst-case Big-O, using the course's conventions in common runtimes (e.g. graph as array-of-linked-lists, vertices enumerable / O(1)-indexable).

KaTeX / math formatting rules (for any authored markdown here)

  • Inline math: $T(i)$, $O(n \log n)$, $G=(V,E)$.
  • Display math fenced in $$ … $$; multi-case with \begin{cases} … \end{cases}.

$$ L(i,j)= \begin{cases} L(i-1,j-1)+1 & \text{if } x_i=y_j,\\ \max\{L(i-1,j),\,L(i,j-1)\} & \text{otherwise.} \end{cases} $$

  • Prefer \operatorname{dist}, \text{capacity}, \le, \ge, \in, \setminus, \emptyset. Avoid unsupported visual-styling commands in reference docs.
