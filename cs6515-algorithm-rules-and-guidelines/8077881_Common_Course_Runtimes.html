Post

When performing runtime analysis for this course, we expect the analysis to be presented as the worst-case runtime analysis, presented in fully simplified Big-O notation.

For example, O(2n + 3n) is expected to be presented as O(n).

As most assessable material for this course is written, this post exists to present the runtimes associated with common actions you may describe in your algorithms.

When no supporting details are presented for special handling in a submission, the default action (underlined) and associated runtimes will prevail.

For the runtimes presented below, n represents the length or size of the object being used.

ListIn the context of the course, we will treat lists as standard arrays, indexable by integers starting at 1.

Creating a list (empty)

  • O(1)

Creating a list with initial contents

  • O(n)

Adding an item to a list (appends to the end)

  • O(1)

Adding an item to a specific non-tail location of a list

  • O(n)

Remove an item from a list (swap tail element and pop)

  • O(1)

Remove an item from the list while maintaining order

  • O(n)

Check the length of a list (via counting)

  • O(n)

Check if a list contains an element (via scanning)

  • O(n)

Hash Table / Hash Set / Map / SetWhile common and practical for software development, these data types have worst-case performance that generally makes them undesirable.

Unless the problem gives you data in this format (typically as a set), you are unlikely to need this.

Creating a hash table (empty)

  • O(1)

Creating a hash table with initial contents

  • O(n^2)

Adding an item to a hash table

  • O(n)

Remove an item from a hash table

  • O(n)

Check the length of a hash table (via counting)

  • O(n)

Check if a hash table contains an element (via scanning)

  • O(n)

GraphIn this course, we will always treat graphs as adjacency lists, with an indexable list of vertices each pointing to a linked list of adjacent vertices. You can assume you know n (number of vertices |V|) and m (number of edges |E|).

Traversing a graph

  • O(n+m)

(Includes reversing, copying, subgraphing, or anything that requires working with all vertices and edges in a graph)

Operating on a single vertex v

  • O(1)

(Includes find, read, removal)

Operating on all (or a subset of) vertices V

  • O(n)

(Includes find, read, removal)

Operating on a single edge e=(u,v)

  • O(n)

  • This is due to the need to traverse up to n vertices in the adjacency list for vertex u, to find and operate on adjacency v.

(Includes find, read, removal)

Operating on all edges |E|

  • O(n+m)

(Includes find, read, removal)

Accessing properties of an edge or vertex

  • O(1)

  • This assumes you already have access to the vertex or edge itself - that is, you have paid the runtime costs above for finding that vertex/edge,

(e.g. weight, color, cost, capacity, etc.)

Special note about simplifications

If you KNOW something about the connectivity of the graph (e.g. MST, Flow, or the problem states it), it is possible to simplify some runtimes:

  • We know the graph is connected and m >= n-1

  • O(n+m) = O(m+m) = O(2m) = O(m)

  • We know the graph is a clique and m = n(n-1)/2

  • O(n+m) = O(n+n^2) = O(n^2)

    • Comment by User 137215 on 2026-06-20 23:45 (Votes: 0)

      You cannot.

      Finding u takes O(1), but finding where v is under u is what takes the time.

      The adjacency list is a list of linked lists, and the endpoints of the edges are stored in the linked list portion, which requires traversal to discover what it contains.

      For a general graph, any given vertex may have at most n-1 edges in its linked list, giving us the worst case runtime for a single edge of O(n) listed above.
